    Meta and Spare Data

    My personal research on the Xbox 360 NAND Spare data, metadata and headers.

    Here i will go into block types and spare data layout.

    Block Types

    There are 2 main block types:

    (Bytes) Big Block Small Block
    Page Data Size 2048 512
    Spare Size 64 16
    Total Page Size 2112 528
    Total Pages 64 32
    Total Blocks 4096 1024
    Block Size 16,896 135,168

    On big block NANDs, for compatibility, the southbridge splits each page into 4 small block sub pages. The NAND dumper (Probably J-Runner with Extras) should've already split the page and interleaved the spare data.

    For 16mb images running an image with a larger FlashFS, such as 17489 devkit conversions, big blocks can be used alongside a filesystems folder on the HDD.

    There is also eMMC (4GB) which has no spare data.

    Small Block Layout

    The 16 end bytes (Spare Data) contain metadata for remapping, FlashFS, and bad block detection.

    Layout Motherboards Block ID Offset Bad Block Marker
    XSB Xenon, Zephyr, Falcon spare[0x0..0x2] spare[0x5]
    PSB/KSB Jasper, Trinity, Corona spare[0x1..0x3] spare[0x5]

    Key Differences: - XSB (Xenon Southbridge): The Logical Block ID is placed at the absolute start of the spare area. - PSB/KSB (Panda & Koala Southbridges): A leading zero byte (0x00) is prepended to the Block ID, shifting it by one byte. This was likely introduced to standardize the spare layout across larger page sizes later.

    Big Block Layout

    Big Block NANDs (256/512MB) utilize a 2048+64 byte physical page structure. While physically larger, the console controller still interacts with the data in 512-byte logical sub-pages.

    Layout Block ID Offset Bad Block Marker
    Big Block spare[0x1..0x3] spare[0x0]

    Spare Data Layout

    While Big Block and Small Block physically append different sizing formats, the overarching protocol uses a 16-byte (0x10) metadata frame attached to each logical 512-byte (0x200) page chunk.

    Within the 16 bytes, the Xbox 360 stores important info for file reconstruction:

    • Block ID (UInt16 LE): Identifies the original physical target.
    • FsSequence (UInt32 BE): An incrementing timestamp order indicating the newest/latest rewrite iteration of a cluster.
    • Bad Block Indicator (UInt8): Sits specifically at index 5 (SmallBlock) or index 0 (BigBlock). 0xFF states the block is clean.
    • FsSize (UInt16 BE)
    • FsPageCount (UInt8)
    • FsBlockType (UInt8): Bits holding what type of data the block contains (like 0x30 or 0x2C for FlashFS Root metadata). Usually piggybacked atop the ECC.

    Bad Blocks

    Unlike normal solid-state setups, the Xbox 360 does not use a FAT-like table or map for tracking bad pages, instead, bad blocks are managed physically:

    • During hardware checks, if a block is deemed corrupt, a flag is dropped directly into its unadulterated spare data frame. For example, in Small Block layouts, if byte 0x05 of the 16-byte spare chunk is 0x00 instead of 0xFF, the block is bad.

    • The replacement block is taken from the hardware reserve sector (the final 32 blocks of your logical partition, typically offset 0x3E0 for 16MB or 0x1E0 for Big Blocks). The physical block ID of the bad block is written into the spare data of the reserve replacement block, establishing an implicit absolute link.
