    NAND Blocks FlashFS

    My personal research on the Small block / Big block NAND Flash File System.

    Allot of what i know here is from RGBuild.

    You should read Spare Data before reading this.

    File System

    The FlashFS is mostly made of a dedicated sector of nand blocks with two intertwined datasets inside the even and odd numbered pages.

    Inside the FlashFS root sector, the block structure changes based on page type.

    Even numbered pages

    Even-numbered pages contain the logical chain map. They act like a classic FAT array, continuous array of uint16 big endian values.

    • Each index basically represents a physical cluster.
    • Value 0x1FFE generally means Unallocated/Free.
    • Value 0x1FFF indicates the Absolute End of a File Chain.

    To read the absolute chain for a file, you find the file's starting block, read that index in the block map, and trace the pointers until it yields 0x1FFF.

    Odd-numbered pages

    Odd-numbered pages inside the root hold the directory listings, known as FileSystemEntry in RGBuild.

    Each entry uses 32 bytes (0x20), structured sequentially without complex nesting:

    Offset Length Type Description
    0x00 0x16 Ascii String The filename, padded with \0. If the first byte is 0x05, the file is marked deleted.
    0x16 0x02 UInt16 (BE) The absolute starting block number linked into the Block Map.
    0x18 0x04 UInt32 (BE) The exact payload size inside the block chain.
    0x1C 0x04 Int32 (BE) Standard DOS-formatted timestamp.

    Creating Files

    When creating or modifying an entry, you:

    1. Find an adjacent sequence of 0x1FFE allocations in the Block Map of adequate size.
    2. Link them together by replacing 0x1FFE with the physical pointers.
    3. Overwrite the odd-paged metadata registry with the new 32-byte directory string referencing the starting block you allocated.
